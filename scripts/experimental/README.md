# Ninja build generators

These scripts generate a `build.ninja` file so you can build cboomer with [Ninja](https://ninja-build.org/) instead of `make`.

You get two implementations that produce the same output: one in Perl (`generate.pl`) and one in Python 3 (`generate.py`). Pick whichever you have installed.

## Usage

```console
# Perl
$ perl scripts/experimental/generate.pl [--dev] [--live] [--mitshm] [--select]
$ ninja

# Python
$ python3 scripts/experimental/generate.py [--dev] [--live] [--mitshm] [--select]
$ ninja
```

Run the generator from the project root (that's where `build.ninja` should end up). The feature flags work the same as `make` and you can combine them, like `--dev --live`.

## How it works

Both scripts do one thing: print a `build.ninja` file with the right variables baked in. They don't contain any build logic themselves. That's all Ninja's job.

### 1. Parse flags

`--dev`, `--live`, `--mitshm`, `--select` each add the matching `-D` define (for example `-DDEVELOPER`). They get concatenated into the `cflags` variable.

### 2. Capture the git hash

Each script runs `git rev-parse HEAD` and writes the result straight into the `cflags` line:

```
cflags = ... -DGIT_HASH=\"abc123def\" ...
```

This is the same thing the Makefile does with `CFLAGS += -DGIT_HASH=\"$(GIT_HASH)\"`. The hash is captured once when you run the generator (configure time), not on every build. The `\"` are shell-escaped quotes so the preprocessor gets the string literal.

### 3. Write the Ninja rules

The generated `build.ninja` defines these rules:

| Rule | Command | Purpose |
|------|---------|---------|
| `mkdir` | `mkdir -p $out` | Create the build directory |
| `gen_shaders` | `scripts/gen_shaders.sh $out` | Convert `.glsl` files to `build/shaders.h` |
| `cc` | `gcc $cflags -MMD -MF $out.d -c -o $out $in` | Compile a `.c` file to a `.o` file |
| `link` | `gcc $cflags -o $out $in $ldlibs` | Link object files into the binary |

The `cc` rule uses `-MMD -MF` to generate Make-style dependency files (`.d` files). Ninja reads those to track `#include` dependencies automatically. If a header changes, the right `.o` files get rebuilt. The Makefile doesn't do this.

### 4. Wire up the build graph

Ninja is told:

1. `build/` must exist before anything else (order-only dep `|| build`)
2. `build/shaders.h` depends on all `.glsl` files and is built with `gen_shaders`
3. `build/main.o` and `build/osd.o` need `build/shaders.h` to exist first (order-only dep)
4. All six `.o` files are linked into the `cboomer` binary

Because dependencies are explicit and `.d` files track headers, Ninja knows exactly what to rebuild when something changes.

## Why a generator script instead of writing build.ninja by hand?

Ninja deliberately has no conditional logic, no shell expansion, and no file-globbing. It expects its input file to be generated by another tool. The generator handles the parts Ninja can't:

- Interpreting `--dev` / `--live` flags
- Running `git rev-parse HEAD` and quoting it correctly
- Listing source files and shader files (so you don't have to maintain two lists)
- Enforcing the right quoting (`\"`) so the shell passes string defines to GCC

This is the same pattern Meson, CMake, and GN use. They all generate `build.ninja` as an intermediate step.

## Differences from `make`

| Aspect | `make` | `ninja` |
|--------|--------|---------|
| Parallel builds | Manual `make -j$(nproc)` | Automatic (uses all cores by default) |
| Header dependencies | Only tracks the Makefile itself | Tracks `.d` files for precise rebuilds |
| Build graph | Inferred from Makefile rules | Explicit `build` statements |
| Configuration | `make dev live` (stacked targets) | `perl generate.pl --dev --live && ninja` |
| Git hash | Evaluated every `make` invocation | Baked at generation time (re-run generator to update) |
